# Нотатки щодо обробки продакшен-збоїв

Дата: 2026-05-04

## Мета

Зафіксувати безпечну стратегію для production-випадків, коли Strapi / GraphQL бекенд тимчасово недоступний.

## Поточна проблема

- Головна сторінка та інші маршрути залежать від серверних `fetch`-запитів до Strapi.
- Кореневий layout теж залежить від Strapi, бо завантажує chrome-дані для header і footer.
- Через це збій бекенда може зламати не лише одну сторінку, а весь shell застосунку.

## Чому редірект на сторінку помилки недостатній

- Простий `redirect('/error')` не є надійним, якщо `app/layout.tsx` також не може відрендеритися без Strapi.
- Якщо сторінка помилки використовує той самий layout, вона може впасти так само, як і звичайна сторінка.
- Редірект змінює URL, але не вирішує реальну проблему доступності shell.
- Якщо продукту потрібен справжній HTTP `503`, редірект не є хорошою заміною коректної outage-обробки.

## Рекомендована стратегія

### 1. Залишити ISR першим рівнем захисту

- У production stale cache кращий за миттєвий редірект на сторінку помилки.
- Якщо є остання успішна версія сторінки, краще віддати її користувачу, ніж одразу ламати UX.
- Це зменшує кількість реальних видимих для користувача збоїв під час коротких проблем із бекендом.

### 2. Додати повністю статичний `app/global-error.tsx`

- Потрібна глобальна аварійна сторінка, яка не залежить від CMS, GraphQL або header/footer data.
- Вона має бути мінімальною, повністю автономною і здатною відрендеритися навіть тоді, коли падає root layout.
- Це основний захист від повного white screen або runtime crash.

### 3. Послабити залежність root layout від обов'язкових CMS-даних

- `app/layout.tsx` не повинен бути single point of failure для всього сайту.
- Header і footer мають підтримувати graceful degradation.
- Варіант A: мати локальний fallback shell.
- Варіант B: мати safe wrapper, який не ламає весь layout, коли Strapi недоступний.
- Головна ідея проста: спрощений shell кращий за повністю зламаний сайт.

### 4. Окремо обробляти збої сторінкових даних

- Для сторінок на кшталт home краще показати fallback UI на зразок "Сервіс тимчасово недоступний", ніж редіректити за замовчуванням.
- Це можна зробити через `app/error.tsx` на рівні сегмента або через контрольований fallback у data layer.
- Такий підхід масштабується краще, ніж побудова редірект-сценаріїв для кожного backend failure.

### 5. Опційно: додати статичний маршрут `/service-unavailable`

- Якщо потрібен окремий outage URL, він має бути повністю незалежним від Strapi.
- Він може бути корисним як запасний UX-сценарій.
- Але він не повинен бути єдиним механізмом захисту.
- Його можна використати для ручних редіректів або edge/server rules, але він не має замінювати `global-error`.

### 6. Якщо важливий справжній `503`, вирішувати це на рівні сервера або CDN

- Якщо бізнес-вимога включає коректний HTTP `503`, це краще реалізовувати на рівні reverse proxy, CDN або custom server.
- Одного лише UI-рівня Next.js недостатньо, щоб надійно гарантувати правильну outage-поведінку.

## Найкращий практичний UX-порядок

- Спочатку: віддати stale ISR page, якщо вона існує.
- Якщо stale page немає: показати статичну глобальну outage-сторінку.
- Якщо shell живий, але конкретна сторінка не може відрендеритися: показати page-level fallback message.
- `/service-unavailable` варто залишити як додатковий інструмент, а не як основу архітектури.

## Рекомендований порядок впровадження

- Додати `app/global-error.tsx`, який не робить жодних fetch-запитів.
- Прибрати жорстку залежність `app/layout.tsx` від обов'язкових CMS-даних.
- Визначити fallback-поведінку для header і footer, якщо Strapi недоступний.
- Додати page-level стратегію для home та інших критичних маршрутів.
- Додати логування та моніторинг backend timeout-ів.
- Перевіряти outage-сценарії через `next build` і `next start`, а не лише через `next dev`.

## Відкриті питання для наступної ітерації

- Чи хочемо ми завжди віддавати stale контент, якщо він існує?
- Який мінімальний shell сайт може показати без Strapi?
- Чи потрібен нам справжній `503`, чи достатньо коректного fallback UI?
- Чи має home мати власну fallback-сторінку, чи нам потрібен один глобальний outage-flow для всього сайту?

## Робочий висновок

Найкращий наступний напрямок такий:

- не будувати рішення навколо редіректу;
- спочатку додати `global-error`;
- потім послабити залежність root layout від CMS;
- після цього додати локальні fallback-сценарії для окремих сторінок.

Такий підхід дасть значно кращу доступність, ніж проста сторінка помилки з редіректом.
