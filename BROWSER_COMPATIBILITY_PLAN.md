# Browser Compatibility Plan

## Мета

Адаптувати поточний Next.js/Tailwind-проєкт під старіші браузери та слабші пристрої настільки, наскільки це можливо, без фанатичного даунгрейду стеку і без відмови від ключових технологій на старті.

Підхід:

1. Спочатку робимо точкові зміни з найбільшим ефектом і найменшим радіусом ризику.
2. Тестуємо сайт після кожного етапу.
3. Лише якщо точкові зміни не дадуть достатнього результату, повертаємось до ідеї даунгрейду Tailwind.

## Поточний контекст

- Стек: `Next.js 16` + `React 19` + `Tailwind CSS 4` + `shadcn/radix`.
- Основне обмеження сумісності зараз задає `Tailwind CSS 4`.
- Найбільші поточні ризики:
  - `oklch()` у базових кольорових токенах
  - modern viewport units (`dvh`, `svh`)
  - `backdrop-filter`, blur, glass-ефекти
  - окремі сучасні CSS-утиліти в shared UI
  - частина browser APIs у client-компонентах

## Принципи виконання

1. Не починати з даунгрейду Tailwind.
2. Не переписувати весь UI вручну без потреби.
3. Спочатку виправляти глобальні токени і shared primitives, а не окремі сторінки.
4. Усе декоративне вважати enhancement, а не критичною базою інтерфейсу.
5. Базовий сценарій успіху: контент, навігація, форми, зображення і ключові інтерактиви працюють навіть там, де частина ефектів деградує.

## Пріоритетний план

### 1. Базова кольорова система

Ціль:

- прибрати найглобальнішу причину проблем із відображенням у старіших браузерах

Що змінюємо:

- переводимо базові semantic tokens з `oklch()` на стабільні `hex`, `rgb()` або `rgba()`
- перевіряємо, чи не зав’язаний критичний вигляд на кольорові утиліти з прозорістю

Основна зона:

- `app/globals.css`

Очікуваний ефект:

- старіші браузери коректно бачать основні кольори фону, тексту, бордерів, поверхонь

Що тестувати після етапу:

- фон сторінки
- читабельність тексту
- кольори header/footer
- кольори карток, секцій, кнопок, форм

### 2. Критичний baseline для тексту, фону і border

Ціль:

- гарантувати, що сайт виглядає цілісно навіть без сучасних декоративних CSS-фіч

Що змінюємо:

- стабілізуємо `body`, `html`, фони секцій, бордери, базові поверхні, основні типографічні кольори
- перевіряємо ключові semantic класи на кшталт `bg-*`, `text-*`, `border-*`

Основна зона:

- `app/globals.css`
- shared layout-обгортки

Очікуваний ефект:

- сайт не виглядає “білим без стилів” або “контрастно зламаним” у межових браузерах

Що тестувати після етапу:

- домашня сторінка
- внутрішні сторінки
- header
- footer
- rich text блоки

### 3. Viewport units і базова висота layout

Ціль:

- уникнути ламання вертикальної композиції на старіших мобільних браузерах

Що змінюємо:

- додаємо або уніфікуємо fallback для `dvh` та `svh` через `vh`
- вирівнюємо підхід у layout, hero і loading-секціях

Основні зони:

- `app/layout.tsx`
- `src/widgets/home/hero/hero-section.tsx`
- `src/widgets/home/home-page-loading.tsx`

Очікуваний ефект:

- перший екран, лоадери і основний shell не “стрибають” і не обрізаються на старіших браузерах

Що тестувати після етапу:

- перший екран на мобільному
- орієнтація portrait/landscape
- відкриття/закриття браузерних панелей на мобільних

### 4. Декоративні ефекти переводимо в enhancement

Ціль:

- зробити так, щоб blur, glass, glow, gradients і напівпрозорості не були критичними для читабельності

Що змінюємо:

- для `backdrop-filter`, blur і подібних ефектів забезпечуємо простий базовий fallback
- там, де потрібно, спрощуємо градієнти і напівпрозорі стани

Основні зони:

- `app/globals.css`
- `src/components/ui/dialog.tsx`
- `src/components/ui/button.tsx`
- home widgets
- navigation widgets

Очікуваний ефект:

- сайт лишається читабельним і візуально акуратним навіть без blur та glass-ефектів

Що тестувати після етапу:

- header у прозорому і скрольнутому стані
- модальні вікна
- кнопки з glass-варіантами
- hero-секції
- картки з напівпрозорими фонами

### 5. Shared UI-примітиви з ризиковими CSS-фічами

Ціль:

- виправити місця, де modern CSS повторюється в багатьох компонентах

Що змінюємо:

- проходимо shared UI і прибираємо або спрощуємо найризиковіші утиліти
- не робимо повний rewrite під Tailwind 3, лише точкові зміни за потреби

Основні зони:

- `src/components/ui/textarea.tsx`
- `src/components/ui/select.tsx`
- `src/components/ui/dialog.tsx`
- `src/components/ui/accordion.tsx`
- `src/components/ui/input.tsx`
- `src/components/ui/button.tsx`

Особлива увага:

- `field-sizing-content`
- modern viewport/value syntax
- blur/fallback states
- складні data-стани, якщо вони дають погану деградацію

Очікуваний ефект:

- форми, селекти, діалоги й акордеони виглядають стабільніше на старіших браузерах

Що тестувати після етапу:

- input/textarea
- select
- dialog
- accordion
- кнопки в різних variants

### 6. Browser APIs і lightweight fallbacks

Ціль:

- не допустити поломки client-функцій через відсутні або слабко підтримувані browser APIs

Що змінюємо:

- додаємо `feature detection` там, де це критично
- за потреби додаємо легкі поліфіли лише для потрібних сценаріїв

Основні зони:

- `src/widgets/search/site-search-dialog.tsx`
- `src/widgets/news/news-list-client.tsx`
- `app/layout.tsx`
- `instrumentation-client.ts`

Особлива увага:

- `AbortController`
- `matchMedia`
- `scrollIntoView({ behavior: "smooth" })`
- `requestAnimationFrame`
- інші дрібні modern APIs, якщо їх зловимо в тестах

Очікуваний ефект:

- пошук, пагінація, скрол до блоку, header-поведінка й інші client-enhancements не падають у старіших браузерах

Що тестувати після етапу:

- пошук по сайту
- пагінація новин
- скрол до секцій
- header on scroll

### 7. Треті сторони і важкі інтерактиви

Ціль:

- перевірити зони, де ризик сумісності вищий через сторонні бібліотеки або складні інтерактиви

Що змінюємо:

- точково адаптуємо проблемні інтерактивні вузли
- не чіпаємо ті місця, які працюють достатньо добре

Основні зони:

- Radix/shadcn overlays
- mobile navigation
- lightbox
- галереї
- каруселі
- сторінки з багатим інтерактивом

Очікуваний ефект:

- інтерактивні сценарії не деградують до повної непрацездатності

Що тестувати після етапу:

- мобільне меню
- відкриття модалок
- галереї і lightbox
- інтерактивні блоки на домашній та внутрішніх сторінках

### 8. Перфоманс для слабких пристроїв

Ціль:

- покращити досвід не лише для старіших браузерів, а й для старіших телефонів

Що змінюємо:

- зменшуємо кількість важких анімацій і паралельних ефектів
- зберігаємо client JS лише там, де він реально потрібен
- за потреби відкладаємо або спрощуємо другорядні ефекти

Основні зони:

- перший екран
- home widgets
- великі інтерактивні секції
- зображення, галереї, lightbox

Очікуваний ефект:

- сайт сприймається швидшим і стабільнішим на слабких пристроях

Що тестувати після етапу:

- стартове завантаження
- плавність скролу
- відкриття меню та модалок
- взаємодія з важкими секціями

## Чекпоінти для прийняття рішень

### Checkpoint A

Після етапів `1-4` оцінюємо:

- чи сайт став візуально стабільним у тестових старіших браузерах
- чи основні проблеми були саме в кольорах, viewport units і декоративних ефектах

Якщо так:

- продовжуємо точкову адаптацію без даунгрейду Tailwind

### Checkpoint B

Після етапів `5-6` оцінюємо:

- чи лишилися системні проблеми через output Tailwind 4
- чи проблем надто багато в shared UI та utility-синтаксисі

Якщо так:

- повертаємось до варіанту з `Tailwind 3.4`

## Що не робимо на старті

- не даунгрейдимо Tailwind одразу
- не переписуємо всі сторінки вручну
- не оптимізуємо дрібні edge-case баги до стабілізації глобальної бази

## Робочий формат

Цю задачу виконуємо в кілька підходів:

1. Беремо наступний найпріоритетніший етап.
2. Вносимо точкові зміни.
3. Запускаємо тестування.
4. Фіксуємо нові спостереження.
5. Вирішуємо, чи рухаємось далі точково, чи переходимо до більш радикального варіанта.

## Поточне рішення

Починаємо з:

1. кольорової системи
2. базового baseline для фону, тексту і borders
3. fallback для viewport units

Даунгрейд Tailwind поки не робимо.
